# Deploy Script Migration Summary

## Overview

The infrastructure deployment has been migrated from plain Terraform to Terragrunt for centralized configuration management.

## Script Comparison

### Key Differences

| Feature | Old deploy.sh (Terraform) | New deploy.sh (Terragrunt) |
|---------|--------------------------|----------------------------|
| **Tool** | Plain Terraform | Terragrunt with Terraform |
| **Configuration** | Separate `terraform.tfvars` per layer | Single `env.hcl` for all layers |
| **Backend Config** | Passed via CLI flags | Generated by Terragrunt |
| **Provider Config** | Static in `versions.tf` | Generated by Terragrunt |
| **Dependencies** | Manual (deploy order) | Automatic (Terragrunt dependency blocks) |
| **Config Layer** | Special case in script | Explicit post-deployment function |
| **Commands** | `terraform init/plan/apply` | `terragrunt init/plan/apply` |

### Feature Parity

Both scripts support the same operations:

- ✅ `deploy` - Deploy all layers or specific layer
- ✅ `plan` - Show deployment plan
- ✅ `validate` - Validate Terraform/Terragrunt configurations
- ✅ `destroy` - Destroy infrastructure
- ✅ `status` - Show deployment status
- ✅ `--layer` - Target specific layer
- ✅ `--auto-approve` - Skip interactive prompts
- ✅ `--dry-run` - Show what would be done
- ✅ `--region` - Override AWS region
- ✅ `--update-ado-secret` - Update ADO PAT in AWS Secrets Manager

### New in Terragrunt Script

1. **Config Layer Improvements**:
   - Explicit `deploy_config_layer()` function
   - Better error handling and validation
   - Interactive prompt after full deployment
   - Comprehensive status checks

2. **Better Dependency Management**:
   - Terragrunt handles dependency graph
   - Automatic output propagation between layers
   - No manual dependency checking needed

3. **Simplified Configuration**:
   - Single `env.hcl` file instead of 3 separate `terraform.tfvars`
   - No need to pass backend config via CLI
   - DRY principle for shared configuration

## Usage Changes

### Old Terraform-Based Script

```bash
# Deploy all layers
./deploy.sh deploy

# Deploy specific layer
./deploy.sh deploy --layer base

# Config layer was automatic (no special handling needed)
```

### New Terragrunt-Based Script

```bash
# Deploy all layers (prompts for config layer)
./deploy.sh deploy

# Deploy specific layer
./deploy.sh deploy --layer base

# Deploy config layer explicitly
./deploy.sh deploy --layer config

# Update ADO PAT
./deploy.sh deploy --layer config --update-ado-secret
```

## Configuration Migration

### Before (Terraform)

Each layer had its own `terraform.tfvars`:

```hcl
# base/terraform.tfvars
cluster_name = "poc-ado-agent-cluster"
vpc_id = "vpc-0555ff8949bb6bb4e"
...

# middleware/terraform.tfvars
(repeated shared values)
...

# application/terraform.tfvars
(repeated shared values)
...
```

### After (Terragrunt)

Single `env.hcl` with all configuration:

```hcl
# env.hcl
locals {
  cluster_name = "poc-ado-agent-cluster"
  vpc_id = "vpc-0555ff8949bb6bb4e"
  
  # All other configuration...
}
```

Each layer's `terragrunt.hcl` references the root config:

```hcl
# base/terragrunt.hcl
include "root" {
  path = find_in_parent_folders()
}

locals {
  env = read_terragrunt_config(find_in_parent_folders("env.hcl"))
}

inputs = merge(
  local.env.locals,
  {
    # Layer-specific overrides
  }
)
```

## Migration Checklist

For teams migrating to the new script:

- [ ] Review `docs/TERRAGRUNT_QUICKSTART.md` for setup instructions
- [ ] Copy `env.sample.hcl` to `env.hcl` and customize with your values
- [ ] Update `TF_STATE_BUCKET` environment variable
- [ ] Test with `./deploy.sh plan` before deploying
- [ ] Run `./deploy.sh deploy` to deploy all layers
- [ ] Run `./deploy.sh deploy --layer config` for post-deployment setup
- [ ] Verify ClusterSecretStore with `kubectl get clustersecretstore`
- [ ] Check External Secrets sync with `kubectl get externalsecrets -A`

## Rollback Plan

If issues arise with the Terragrunt-based script:

1. **Reference Legacy Script**: `docs/deploy-legacy.sh` is available
2. **Use Original tfvars**: Each layer still has its `terraform.tfvars` files
3. **Direct Terraform**: Can run `terraform` commands directly in each layer

## Benefits of Migration

1. **Single Source of Truth**: `env.hcl` replaces 3 separate `terraform.tfvars` files
2. **Automatic Dependencies**: Terragrunt manages layer dependencies automatically
3. **Better Config Management**: Centralized configuration reduces duplication
4. **Easier Multi-Environment**: Copy `env.hcl` for each environment (dev, staging, prod)
5. **Consistent Backend Config**: Generated automatically, no CLI flags needed
6. **Improved Maintainability**: DRY principle reduces configuration drift

## Known Issues & Limitations

### Current

- Config layer must be run after all Terraform layers (by design)
- Terragrunt caches need to be cleaned manually if configuration changes (`rm -rf .terragrunt-cache`)
- Environment-specific configurations require separate `env.hcl` files

### Future Improvements

- [ ] Add `terragrunt run-all` support to deploy all layers in one command
- [ ] Implement config layer as Terragrunt module
- [ ] Add automated rollback on deployment failures
- [ ] Create GitHub Actions workflow for CI/CD
- [ ] Add support for multiple environments in single repository
